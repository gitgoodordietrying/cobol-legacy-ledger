## Supplement A: Batch Processing Trace Format

**Why this matters:** The terminal output during batch processing IS the demo. When the 15-minute presentation shows transactions flowing through TRANSACT.cob, the audience sees this output. A financial industry reviewer has watched thousands of batch job logs scroll by on green-screen terminals. If ours looks right, they'll trust the whole project. If it looks like a Python print statement, they won't.

**Where this goes:** TRANSACT.cob's DISPLAY statements during batch mode. The Python bridge captures this stdout and can either relay it directly to the console or parse it for the frontend.

### The Format

TRANSACT.cob must produce this exact output structure during batch processing:

```

  NODE: BANK_A — FIRST NATIONAL BANK
  DATE: 2026-02-17  TIME: 22:45:03
  INPUT: BATCH-INPUT.DAT
========================================================

--- BEGIN BATCH RUN ---

SEQ  STATUS  TYPE  AMOUNT        ACCOUNT     BALANCE-AFTER  DESCRIPTION
---  ------  ----  -----------   ----------  -------------  --------------------------------
001  OK      DEP   $  5,000.00   ACT-A-001   $ 20,420.50   Payroll deposit — Santos
002  OK      WDR   $    750.00   ACT-A-003   $  4,467.25   ATM withdrawal — Chen
003  OK      DEP   $  9,500.00   ACT-A-006   $ 21,730.00   Cash deposit — Elena Petrov
 ** COMPLIANCE NOTE: Deposit $9,500.00 within $500 of $10,000 CTR threshold

006  OK      XFR   $  3,200.00   ACT-A-005   $ 12,400.00   Transfer to ACT-A-007 — Park
```

### Implementation Notes for Claude Code

**Column alignment matters.** Use COBOL's PIC editing to right-justify dollar amounts. The `$` sign and comma formatting come from `PIC $$$,$$$,$$9.99` — this is a COBOL strength. Let COBOL do the formatting, not Python.

**Status codes map to the COMCODE.cpy values:**
| Display Code | COMCODE Value | Meaning |
|-------------|---------------|---------|
| `OK` | `RC-SUCCESS` / `'00'` | Processed |
| `FAIL01` | `RC-NSF` / `'01'` | Insufficient funds |
| `FAIL02` | `RC-LIMIT-EXCEEDED` / `'02'` | Daily limit exceeded |
| `FAIL03` | `RC-INVALID-ACCT` / `'03'` | Account not found |
| `FAIL04` | `RC-ACCOUNT-FROZEN` / `'04'` | Account frozen |
| `FAIL99` | `RC-FILE-ERROR` / `'99'` | File I/O error |

**The compliance note** is generated by TRANSACT.cob itself, not by the AI layer. This is important — the COBOL code contains the business rule `IF TX-AMOUNT > 9500 AND TX-TYPE = 'D'` and produces the warning. The Python bridge just relays it. In a real bank, this flag would route to a compliance queue. In our demo, it's a visible talking point: "The COBOL catches this. Not the AI. The AI just explains why it matters."

**The summary footer** is produced by TRANSACT.cob after processing all batch records. Running totals are maintained in WORKING-STORAGE accumulators. This is classic COBOL batch reporting — the machine was born to do this.

**The hash line** is the one element NOT produced by COBOL. The Python bridge appends it after syncing the batch to the integrity chain. Mark it in the output: the last two lines come from Python, everything above comes from COBOL. This boundary is the whole thesis.

### DISPLAY Implementation Pattern

In TRANSACT.cob, the batch processing loop should look like:

```cobol
       PROCESS-ONE-TRANSACTION.
      *    (validation, balance update, file rewrite happen here)
      *    Then format the trace line:
           ADD 1 TO WS-BATCH-SEQ
           MOVE WS-BATCH-SEQ TO WS-SEQ-DISPLAY

           EVALUATE WS-RESULT-CODE
               WHEN '00'
                   MOVE "OK    " TO WS-STATUS-DISPLAY
               WHEN '01'
                   MOVE "FAIL01" TO WS-STATUS-DISPLAY
               WHEN '04'
                   MOVE "FAIL04" TO WS-STATUS-DISPLAY
               WHEN OTHER
                   STRING "FAIL" WS-RESULT-CODE
                       DELIMITED SIZE INTO WS-STATUS-DISPLAY
                   END-STRING
           END-EVALUATE

           DISPLAY WS-SEQ-DISPLAY "  "
               WS-STATUS-DISPLAY "  "
               WS-TYPE-DISPLAY "  "
               WS-AMOUNT-DISPLAY "   "
               WS-IN-ACCT-ID "  "
               WS-BALANCE-DISPLAY "  "
               WS-IN-DESCRIPTION

      *    Compliance flag — COBOL generates this, not Python
           IF WS-IN-TYPE = 'D'
               AND WS-IN-AMOUNT > 9500.00
               DISPLAY " ** COMPLIANCE NOTE: Deposit "
                   WS-AMOUNT-DISPLAY
                   " within $500 of $10,000 CTR threshold"
           END-IF.
```

### Edited PIC Clauses for Formatted Output

Add these to TRANSACT.cob's WORKING-STORAGE:

```cobol
       01  WS-DISPLAY-FIELDS.
           05  WS-SEQ-DISPLAY      PIC ZZ9.
           05  WS-STATUS-DISPLAY   PIC X(6).
           05  WS-TYPE-DISPLAY     PIC X(4).
           05  WS-AMOUNT-DISPLAY   PIC $$$,$$$,$$9.99.
           05  WS-BALANCE-DISPLAY  PIC $$$,$$$,$$9.99.
```

The `Z` suppresses leading zeros. The `$` floats. The `,` inserts commas. This is COBOL doing what COBOL was designed to do — formatted business reporting. Let it shine.

## Supplement B: KNOWN_ISSUES.md Template

**Why this matters:** Every real legacy system has known limitations that are documented, accepted, and managed rather than fixed. This is the difference between a student project ("I couldn't figure out dynamic arrays") and an engineering decision ("We use fixed-size arrays because that's how COBOL works in production and it's adequate for the account volumes we handle"). An Edward Jones interviewer will immediately recognize this as mature systems thinking.

**Where this goes:** Root of the repository, alongside README.md.

### The File

```markdown
# Known Issues and Intentional Limitations

**Project:** cobol-legacy-ledger
**Last Updated:** 2026-02-XX

This document catalogs known limitations in the COBOL programs and
Python integration layer. These are engineering decisions, not bugs.
Each reflects either an authentic COBOL constraint preserved for
realism, or a scope boundary appropriate to the demonstration context.

---

## COBOL Limitations (Authentic — By Design)

### A1: Fixed Account Table Size (100 accounts per node)

**What:** ACCOUNTS.cob loads all accounts into a WORKING-STORAGE
`OCCURS 100 TIMES` array. Nodes with more than 100 accounts would
overflow silently.

**Why it's kept:** COBOL does not have dynamic memory allocation.
Production COBOL systems handle this with either compile-time table
sizing (common in the 1980s–2000s) or indexed file organization
(VSAM/ISAM). Our demonstration uses sequential files and in-memory
tables because this is the pattern most legacy banking systems
actually use for branch-level account volumes.

**Risk:** Zero. Our largest node (BANK_A, BANK_C, BANK_E) has 8
accounts. The 100-slot ceiling is 12x headroom.

**Production fix:** Replace sequential file I/O with indexed
organization (`ORGANIZATION IS INDEXED, ACCESS MODE IS DYNAMIC,
RECORD KEY IS ACCT-ID`) for random access without full-file loading.

---

### A2: Sequential Account Search (O(n) per lookup)

**What:** FIND-ACCOUNT performs a linear scan of the in-memory
account table. Lookup time is proportional to account count.

**Why it's kept:** This is how COBOL programs search in-memory
tables when the data set is small. The `SEARCH ALL` verb exists
for binary search on sorted tables, but it requires `ASCENDING KEY`
declarations and pre-sorted data — complexity not justified for
8 accounts per node.

**Risk:** Negligible. At 8 accounts, the loop runs 8 iterations.
At 100 accounts (our ceiling), it would run 100 — still
sub-millisecond.

**Production fix:** Use `SEARCH ALL` with `ASCENDING KEY IS
WS-A-ID` and ensure accounts are loaded in sorted order, or
switch to indexed file organization for disk-level keyed access.

---

### A3: Full-File Rewrite on Every Modification

**What:** When any account record changes (balance update, status
change), the program rewrites the entire ACCOUNTS.DAT file from
the in-memory array. This is `OPEN OUTPUT` (truncate) followed by
`WRITE` for every record.

**Why it's kept:** Sequential file organization does not support
random record updates. This is the standard pattern in COBOL
systems that use flat files instead of indexed files (VSAM). Many
production batch systems still operate this way — the nightly batch
reads the master file, applies all transactions, and writes a new
master file. Our per-transaction rewrite is a simplification of
this pattern.

**Risk:** Low. File sizes are small (< 7 KB per node at 100
accounts × 70 bytes). The rewrite completes in under 1ms. In a
concurrent environment this would be a race condition; our
single-threaded batch model avoids this.

**Production fix:** Use indexed file organization with `REWRITE`
verb for in-place record updates, or use a database (DB2, IMS)
for concurrent access.

---

### A4: No Transaction Rollback

**What:** If TRANSACT.cob updates the account balance but then
fails to write the transaction record (e.g., disk full), the
account file reflects the change but the transaction log does not.
There is no two-phase commit.

**Why it's kept:** COBOL flat-file systems do not have
transactional semantics. This is a real limitation of the
technology. Production systems handle this through batch
reconciliation — if the transaction log and account balances
disagree at end-of-day, the discrepancy is flagged for manual
review. Our integrity chain serves a similar purpose: if the
chain entry count doesn't match the transaction count, something
was lost.

**Risk:** Medium. A crash between the account write and
transaction write could leave inconsistent state. Mitigated by:
(1) the integrity chain detects the discrepancy during verify,
and (2) our demonstration runs on reliable local storage, not
1980s magnetic tape.

**Production fix:** Transaction processing monitors (CICS, IMS/TM)
provide two-phase commit across multiple file updates. Modern COBOL
accesses databases (DB2) with full ACID guarantees.

---

### A5: Date Handling Is Not Y2K-Safe (Acknowledged)

**What:** Date fields use `PIC 9(8)` storing YYYYMMDD as a plain
integer. There is no date validation — `99991301` would be accepted
as a valid date.

**Why it's kept:** Adding date validation logic would add 50+ lines
per program for a feature not central to the demonstration. The
YYYYMMDD format avoids the original Y2K issue (2-digit years) but
doesn't validate ranges.

**Risk:** Zero in demo context. All dates are hardcoded or
system-generated.

**Production fix:** Use intrinsic functions (`FUNCTION
WHEN-COMPILED`, `FUNCTION CURRENT-DATE`) and validate with
`FUNCTION INTEGER-OF-DATE`.

---

## Python Integration Layer Limitations

### B1: Bridge Assumes Single-Threaded COBOL Execution

**What:** The Python bridge calls COBOL binaries via subprocess
and assumes exclusive access to .DAT files during execution.

**Why it's kept:** COBOL flat-file I/O is inherently
single-threaded. This is the real constraint.

---

### B2: Balance Parsing Depends on GnuCOBOL Display Format

**What:** The bridge parser for `PIC S9(10)V99` values depends
on GnuCOBOL's specific display rendering (sign overpunch
characters, leading zeros). A different COBOL compiler might
render the same PIC differently.

**Why it's kept:** This is a known GnuCOBOL-specific coupling.
The smoke test (Section 0.1 of COBOL Style Reference) exists
specifically to verify the display format before the parser
is written.

---

## Scope Limitations (By Design)

### C1: No Multi-User Concurrent Access

Demonstration context. Single operator, sequential batch.

### C2: No Encrypted Data at Rest

.DAT files are plaintext. The integrity chain detects tampering
but does not prevent reading. Production systems encrypt sensitive
fields or use encrypted storage volumes.

### C3: No Network Communication Between Nodes

All 6 nodes run on the same machine with separate data directories.
Inter-bank transfers are coordinated by the Python settlement layer,
not by COBOL-to-COBOL communication. This is actually realistic —
real inter-bank settlement uses a central coordinator (Fed, ACH,
SWIFT), not direct bank-to-bank messaging.

---

## How to Read This Document

Each known issue follows the same structure:

1. **What** — objective description of the limitation
2. **Why it's kept** — the engineering reasoning
3. **Risk** — honest assessment for the demo context
4. **Production fix** — what a real team would do at scale

This structure demonstrates that the developer understands both
the limitation and the production path. The limitations are
preserved because they're authentic to the technology, not because
the developer couldn't solve them.
```

## Supplement C: Program Header Block Template

**Why this matters:** Real production COBOL has extensive header blocks. When a financial industry reviewer opens one of our `.cob` files, the first thing they see should feel familiar — like opening a program that's been in production since 1987 with a change log maintained by three generations of developers.

**Where this goes:** Every `.cob` file. Claude Code should use this template exactly.

### The Template

```cobol
      *================================================================*
      *  Program:     ACCOUNTS.cob
      *  System:      LEGACY LEDGER — Account Lifecycle Management
      *  Node:        All (same binary, per-node data directories)
      *  Author:      AKD Solutions
      *  Written:     2026-02-XX
      *  Modified:    2026-02-XX
      *
      *  Purpose:
      *    Account master file CRUD operations. Creates, reads,
      *    updates, and lists customer and nostro accounts stored
      *    in the node's ACCOUNTS.DAT sequential file.
      *
      *  Operations (via command-line argument):
      *    CREATE  — Add new account to master file
      *    READ    — Display single account by ID
      *    LIST    — Display all active accounts
      *    UPDATE  — Modify account name, type, or status
      *    CLOSE   — Set account status to 'C' (closed)
      *    FREEZE  — Set account status to 'F' (frozen)
      *
      *  Files:
      *    Input/Output: ACCOUNTS.DAT (LINE SEQUENTIAL, 70-byte records)
      *
      *  Copybooks:
      *    ACCTREC.cpy  — Account record layout (70 bytes)
      *    COMCODE.cpy  — Shared status codes and bank identifiers
      *
      *  Output Format (to STDOUT, pipe-delimited):
      *    Success: OK|OPERATION|ACCT-ID|field|field|...
      *    Account: ACCOUNT|ACCT-ID|NAME|TYPE|BALANCE|STATUS|OPENED|LASTACT
      *    Error:   ERROR|REASON|detail
      *    Result:  RESULT|XX  (where XX = status code from COMCODE.cpy)
      *
      *  Exit Codes:
      *    RESULT|00 — Success
      *    RESULT|03 — Account not found (or duplicate on CREATE)
      *    RESULT|04 — Account frozen (cannot modify)
      *    RESULT|99 — File I/O error
      *
      *  Dependencies:
      *    Requires ACCOUNTS.DAT to exist in CWD (working directory).
      *    CWD is set by the Python bridge to banks/{NODE}/.
      *    If file does not exist, returns RESULT|99 on READ/LIST,
      *    or creates it on first CREATE.
      *
      *  Change Log:
      *    2026-02-XX  AKD  Initial implementation
      *
      *================================================================*
```

### What Makes This Authentic

1. **The `System:` line** — production COBOL always identifies which system the program belongs to. This connects individual programs to the larger banking system.

2. **The `Node:` line** — documents the multi-node architecture. A reviewer immediately understands this binary is shared across nodes.

3. **The `Operations:` block** — lists every operation the program handles. This is the external API contract. The Python bridge developer reads this to know what command-line arguments to pass.

4. **The `Output Format:` block** — documents the pipe-delimited protocol. This is the interface contract between COBOL and Python. If this doesn't match the bridge parser, the system breaks.

5. **The `Exit Codes:` block** — maps to COMCODE.cpy values. This is how COBOL communicates status to the calling system.

6. **The `Change Log:`** — even with one entry, this signals "this program is maintained." In a real system, this log would have 30 entries spanning 20 years.

### Per-Program Customization

Each of the four programs gets the same template structure with different content:

**TRANSACT.cob** — Add: Operations include `DEPOSIT`, `WITHDRAW`, `TRANSFER`, `BATCH`. Files include both `ACCOUNTS.DAT` and `TRANSACT.DAT`. Note that `BATCH` reads from `BATCH-INPUT.DAT`. Document the batch trace format (Supplement A).

**VALIDATE.cob** — Add: Called by `TRANSACT.cob` before debit operations. Not invoked directly by the Python bridge. Document the validation sequence: check account exists → check account active → check sufficient balance → check daily limit.

**REPORTS.cob** — Add: Operations include `STATEMENT`, `LEDGER`, `EOD`, `AUDIT`. All operations are read-only — no file modifications. Document each report's output format.

## Supplement D: What to Put in the README's "About the COBOL" Section

This isn't a file for Claude Code — it's the narrative framing that makes the COBOL impressive to a non-COBOL audience. Add a section in the README between the architecture diagram and the quick start:

### Talking Points

**"These are real COBOL programs, not wrappers."** ACCOUNTS.cob manages the account master file. TRANSACT.cob processes deposits, withdrawals, and transfers with balance validation. REPORTS.cob generates ledger summaries and account statements. VALIDATE.cob enforces business rules including regulatory threshold detection. Every program reads and writes fixed-width sequential files — the same I/O pattern used in production banking systems since the 1960s.

**"The COBOL does the banking. Python does the plumbing."** The Python bridge doesn't reimplement any banking logic. It launches COBOL binaries as subprocesses, passes operations via command-line arguments, parses the pipe-delimited output, and syncs the results to SQLite for the monitoring console. If you deleted all the Python, the COBOL programs would still process transactions correctly — you'd just be reading .DAT files by hand.

**"The limitations are documented, not hidden."** See `KNOWN_ISSUES.md`. Fixed-size account tables, sequential search, full-file rewrites, no transaction rollback — these are real COBOL constraints preserved for authenticity. Each one includes the production fix that a real modernization team would implement. The document demonstrates that we understand both the limitation and the path forward.

**"The batch trace log is COBOL output, not Python formatting."** When you watch the batch demo, the columnar transaction report with formatted dollar amounts, status codes, and compliance flags — that's COBOL's DISPLAY output piped through to your terminal. COBOL's `PIC $$$,$$$,$$9.99` edited fields do the dollar formatting. COBOL's business rules generate the CTR compliance warnings. Python just relays it.
