# Claude Code Configuration: cobol-legacy-ledger

## Project Overview

Educational COBOL teaching resource — a fully functional inter-bank settlement system demonstrating COBOL to software engineers. Non-invasive cryptographic integrity layer wraps unmodified COBOL with Python observation and SHA-256 hash chain verification. 6 nodes (5 banks + clearing house).

**Narrative**: "COBOL isn't the problem. Lack of observability is."

---

## Project Purpose

This is a **teaching resource** for IT classes. Every COBOL and Python source file contains thorough educational comments explaining syntax, patterns, and design decisions. The documentation serves as a manual, glossary, and textbook.

---

## Critical Principles

1. **COBOL Immutability** — Wrap, don't modify. Legacy systems unchanged.
2. **Cryptographic Integrity** — SHA-256 chain + HMAC. Detect tampering in <100ms.
3. **Per-Node Database** — Each node has its own SQLite DB. No shared ledger.
4. **6-Node Architecture** — Fixed: BANK_A, BANK_B, BANK_C, BANK_D, BANK_E, CLEARING.
5. **Production COBOL** — Headers + KNOWN_ISSUES.md documentation required.
6. **Testability** — Every requirement must be testable. 807 tests, all green.
7. **No Node.js** — Static HTML/CSS/JS only. No npm, no build process. Web console served via FastAPI StaticFiles at `/console/`.
8. **Clear Error Paths** — Status codes: 00=success, 01=NSF, 02=limit, 03=invalid, 04=frozen, 99=error.
9. **Educational Comments** — Every source file teaches concepts inline with `COBOL CONCEPT:` blocks. Layer 3 (API/LLM) files follow the same standard: 20-40 line module docstrings, `# ── Title ─────────` section banners, class/method docstrings, and inline comments on non-obvious lines.

### Record Formats

**ACCTREC (Account)**: 70 bytes
- ACCT-ID (10) | ACCT-NAME (30) | ACCT-TYPE (1) | ACCT-BALANCE (12, implied decimal) | ACCT-STATUS (1) | ACCT-OPEN-DATE (8) | ACCT-LAST-ACTIVITY (8)

**TRANSREC (Transaction)**: 103 bytes
- TRANS-ID (12) | TRANS-ACCT-ID (10) | TRANS-TYPE (1) | TRANS-AMOUNT (12, implied decimal) | TRANS-DATE (8) | TRANS-TIME (6) | TRANS-DESC (40) | TRANS-STATUS (2) | TRANS-BATCH-ID (12)

**BATCH-INPUT**: Pipe-delimited
- Format: `ACCOUNT_ID|TYPE|AMOUNT|DESCRIPTION` (D/W/I/F)
- Format: `ACCOUNT_ID|T|AMOUNT|DESCRIPTION|TARGET_ID` (transfers only)

## Troubleshooting

**Q: Where do I start?**
A: Read `COBOL-BANKING/src/SMOKETEST.cob` — the simplest program, teaches all 4 COBOL divisions. Then follow the [Learning Path](docs/LEARNING_PATH.md).

**Q: What if cobc isn't installed?**
A: The system works without COBOL installed. build.sh skips compilation, seed.sh uses Python file I/O (Mode B). All unit tests pass in Mode B (E2E tests require a running server).

**Q: Why per-node databases?**
A: Financial systems are distributed. Each node operates independently. Per-node isolation mirrors real banking — distributed, independent, reconciled through settlement.
